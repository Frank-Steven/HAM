# 0xFF HAM 3.0 概述

## HAM 的由来

HAM 最初是为了设计 HoundWhiteboard 的脚本化操作而诞生的项目，但随着 HAM 的发展，逐渐成为独立的编程语言设计。

HAM 的设计过程受到函数式编程范式的重度影响，决定了 HAM 是一门函数式编程语言。HAM 的设计遵循规则最小化原则，这意味着 HAM 不会大量引入语法糖，而是注重用原始逻辑来建构复杂的系统。

然而，真正实现一门编程语言对于一个人来说实在比较困难，因此 HAM 目前没有任何实现，实在是遗憾。。。

## 既然 HAM 没有实现，你为什么还在读这个“指南”？

1. HAM 语言严格遵循函数式编程范式，这意味着你可以通过了解 HAM 来了解函数式；

2. HAM 的函数式和广泛流传的函数式工程范式有些许差异，通过跟随 HAM 的思想可以有一种不同的对函数式的见解，这种见解更加干净，没有很多条条框框的限制；

3. 你可能会成为 HAM 的贡献者；

4. HAM 的逻辑不冗余，这迫使你彻底放弃命令式的思维；

5. 没有什么理由，你单纯喜欢 HAM。

## HAM 的演化史

### HAM 1.0

这个版本的 HAM 意外地非常像简化版的 Scala，有基本的表达式表示法。

### HAM 2.0

由于上一个版本的语法太过于繁复，这个版本主要优化了语法，并引入默认多值返回（或者叫多语义返回）的机制，并对 HAM 的工程原理、编译原理进行了一些梳理。

### HAM 3.0

上一个版本逻辑混乱，并且为了所谓的“工程意义”做了太多妥协—— HAM 2.0 引入了太多的外部控制逻辑，对于多语义返回的逻辑推导也有自相矛盾，在 3.0 版本进行了重塑，替换为了清晰简单的逻辑结构体系。

## HAM 总览

整个 HAM 的语法体系由两个基本的结构构成：

1. 组合结构
2. 表达式

其中组合结构是 HAM 中语义的声明格式，表达式是 HAM 的逻辑基础。
